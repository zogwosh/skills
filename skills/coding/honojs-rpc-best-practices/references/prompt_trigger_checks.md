# Hono RPC Best Practices Prompt Trigger Checks

Use these checks when changing the skill description, trigger boundaries, or examples.

## Positive prompts

These prompts should trigger this skill:

- "Refactor this Hono app so `hc<AppType>` keeps route and response inference."
- "Review this Hono RPC module for validator contract issues."
- "The client generated by `hono/client` no longer infers the request body. Find the route typing problem."
- "Choose between `hono/validator`, `@hono/zod-validator`, and `@hono/standard-validator` for this RPC route."
- "Split this Hono route file without losing `AppType` inference."

## Negative prompts

These prompts should not trigger this skill:

- "Design generic REST conventions for Express."
- "Build a frontend-only React form."
- "Review these Zod schemas without any Hono route or client context."
- "Explain OAuth middleware for a non-Hono server."
- "Write OpenAPI docs for a service that does not use Hono RPC."

## Boundary checks

- Trigger when the task mentions Hono plus RPC, `hc`, `AppType`, route inference, validator middleware, or `hono/client`.
- Do not trigger for validation work that is independent of Hono route contracts.
- Do not trigger for generic Hono middleware questions unless RPC typing, validator contracts, or client inference is part of the request.
